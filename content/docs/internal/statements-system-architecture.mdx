## Overview

The **Statements** system is the wiki's canonical structured assertion layer. It stores discrete facts and attributed claims about entities in a typed, temporal, citeable format.

Statements replace the legacy **Claims** system (1,514 records in PostgreSQL, 89% with quality issues) and complement the **Facts** system (139 volatile numeric values in YAML). This document is the single reference for the system's architecture, design decisions, and roadmap.

## Data Model

### Three Tables

**`statements`** — the core assertion store:

| Column | Type | Purpose |
|--------|------|---------|
| `id` | bigserial | Primary key |
| `variety` | text | `"structured"` or `"attributed"` |
| `subject_entity_id` | text (FK) | What entity this statement is about |
| `property_id` | text (FK) | Links to controlled vocabulary (structured only) |
| `value_numeric`, `value_text`, `value_entity_id`, `value_date`, `value_series` | various | Typed value columns (structured only) |
| `value_unit` | text | Display hint (e.g., "USD", "percent") |
| `qualifier_key` | text | Context (e.g., "round:series-g") |
| `valid_start`, `valid_end` | text | Temporal validity ("2026-02", null = still true) |
| `statement_text` | text | Free-text assertion (attributed variety) |
| `attributed_to` | text (FK) | Entity making the claim (attributed only) |
| `verdict`, `verdict_score`, `verdict_quotes`, `verdict_model` | various | Verification metadata |
| `status` | text | `"active"`, `"superseded"`, `"retracted"` |
| `source_fact_key` | text | Traceability link to YAML facts (e.g., "anthropic.6796e194") |

**`properties`** — the controlled vocabulary:

| Column | Type | Purpose |
|--------|------|---------|
| `id` | text PK | kebab-case identifier (e.g., "valuation", "ceo") |
| `label` | text | Human-readable name |
| `category` | text | financial, organizational, safety, performance, milestone, relation |
| `value_type` | text | "number", "string", "entity", "date" |
| `unit_format_id` | text | References `UNIT_FORMATS` TS constant for display |
| `staleness_cadence` | text | "quarterly", "annually", null |

**`statement_citations`** — multi-source provenance:

| Column | Type | Purpose |
|--------|------|---------|
| `statement_id` | bigint (FK) | Which statement this cites |
| `resource_id` | text (FK) | Links to resources table |
| `url` | text | Fallback if no resource |
| `source_quote` | text | Excerpt from the source |
| `is_primary` | boolean | Which source to show first |

### Two Varieties

**Structured** statements have a subject entity, a property from the controlled vocabulary, and typed value columns. They represent facts the wiki asserts:

```
Subject: Anthropic
Property: valuation
Value: 380,000,000,000 (USD)
Valid: 2026-02
Status: active
```

**Attributed** statements report what a specific person or publication said. They use `statement_text` (free-text) and `attributed_to` instead of structured value columns:

```
Subject: Anthropic
Attributed to: Dario Amodei
Text: "I think within one to two years, Claude will be performing at the level of a Nobel Prize winner"
Valid: 2025-11
Status: active
```

## Key Design Decisions

These decisions were validated during the Claims era and carry forward to Statements:

### 1. Multi-Source Citations (Join Table)

A single statement can be attested by multiple independent resources. "Anthropic raised a Series G" appears in Reuters, TechCrunch, Bloomberg, and Anthropic's own press release. The `statement_citations` join table (not a JSONB array) enables per-source quotes, primary source flagging, and independent source verdicts.

### 2. Faceted Classification

Five orthogonal facets with fixed vocabulary, enforced at write time:

| Facet | Values |
|-------|--------|
| **Variety** | structured, attributed |
| **Status** | active, superseded, retracted |
| **Category** (via property) | financial, organizational, safety, performance, milestone, relation |
| **Verdict** | verified, unverified, unsupported, disputed |
| **Temporal granularity** | year, quarter, month, day |

Tags are avoided — they degrade past \~1,000 items without governance.

### 3. Temporal Validity

`valid_start` and `valid_end` are text columns (not dates) to support partial dates like "2025", "2025-07", "2026-02" from YAML facts. `valid_end IS NULL` means "currently believed true." When a value changes, the old statement gets `status: superseded` and `valid_end` set; a new statement is created.

### 4. Property Taxonomy

Properties are a controlled vocabulary seeded from `data/fact-measures.yaml`. Each property has a `value_type` (number, string, entity, date), optional unit formatting, and staleness cadence. This ensures consistent querying — "show me all entities with a valuation" works because `valuation` is a known property.

## Relationship to Facts (YAML)

**Facts** and **Statements** coexist as complementary systems:

| | Facts | Statements |
|---|---|---|
| **Storage** | YAML (`data/facts/*.yaml`) | PostgreSQL |
| **Scope** | Volatile numeric values only | All assertion types |
| **Rendering** | `\<F>` / `\<Calc>` components | Statements UI, entity pages |
| **Count** | 139 facts, 18 entities | Growing |
| **Update mechanism** | Edit YAML → rebuild | API / extraction pipeline |

**Bridge pattern**: Statements link to Facts via `source_fact_key` (e.g., "anthropic.6796e194"). This provides traceability without requiring migration of the YAML → `\<F>` → build-time pipeline. Facts are the curated, display-ready promotion of important numeric statements.

**Long-term**: Facts may eventually become a subset of Statements (numeric statements with `is_canonical: true`), but the YAML-based `\<F>` component pipeline works well and doesn't need immediate migration. See <EntityLink id="E886" name="fact-system-strategy">Fact System Strategy (E886)</EntityLink>.

## Relationship to Claims (Legacy)

The **Claims** system is the predecessor. Key differences:

| | Claims (legacy) | Statements (current) |
|---|---|---|
| **Schema** | 40+ fields, many unused | Lean: typed value columns + property taxonomy |
| **Classification** | 7 claim_types, claim_mode, claim_category, inference_type | 2 varieties (structured/attributed) + property categories |
| **Source tracking** | `claim_sources` join table + `resourceIds` JSONB (redundant) | `statement_citations` join table only |
| **Quality** | 89% have issues (see <EntityLink id="E901">E901</EntityLink>) | Clean-slate with validation |
| **UI** | `/claims/*` (10 routes) | `/statements/*` (6 routes) |

**Migration status**: The 1,514 existing claims are not yet migrated to Statements. Options:
1. Re-extract from source pages using improved extraction prompts
2. Mechanical migration with quality fixes applied
3. Selective migration (only high-quality claims worth preserving)

The `/claims` UI routes remain accessible but are considered legacy.

## Known Quality Issues

From the <EntityLink id="E901" name="claims-data-quality-audit">Claims Data Quality Audit (E901)</EntityLink> — these apply to any assertion extraction system and inform Statements validation:

| Issue | Prevalence | Mitigation |
|-------|-----------|------------|
| Missing related entities | 62.5% | Enforce entity linking at extraction |
| Unsourced assertions | 58.6% | Require at least one citation |
| Not self-contained | 45.8% | Extraction prompt: include entity name |
| Relative phrase starts | 13.1% | Post-processing validation |
| Exact duplicates | 10.0% | Dedup before insertion |
| Wrong entity attribution | 5.2% | Multi-pass extraction with entity routing |

## Open Design Questions

Carried forward from the Claims era, not yet resolved for Statements:

### 1. Global vs Per-Page Statements

Should the same fact appear as one global record (with page references) or one record per page? Global is cleaner for queries but requires deduplication infrastructure. Per-page is simpler but inflates volume 3-5x.

**Current**: De facto per-entity (each statement belongs to one subject entity). Cross-page dedup is not implemented.

### 2. Extraction Direction

**From pages → statements** (current approach) vs **from resources → statements** (higher value, harder). Resource-first extraction means claims about Neel Nanda come from his papers and interviews, not from our wiki prose about him. This is the right long-term direction for person/org pages.

### 3. Verification Economics

Full verification costs \≈\$0.01-0.05 per statement. At 700 pages \* 100 statements/page = 70,000 statements, a full pass costs \$700-3,500. The system must decide: verify everything lazily? Selectively? On insertion only?

### 4. Endorsed vs Attributed Boundary

The structured/attributed distinction is clear for "Anthropic's valuation is X" (structured) vs "Dario says Y" (attributed). It's fuzzy for "Research suggests X" or "Anthropic has strong safety commitments." The extraction pipeline needs consistent classification rules.

## Roadmap

### Phase 1: Foundation (current)

- Statements schema deployed and seeded from YAML facts
- `/statements` UI with browse, filter, entity views
- Property taxonomy established (~25 properties)
- `source_fact_key` bridge to YAML facts

### Phase 2: Extraction Pipeline

- Build `crux statements extract` (analogous to `crux claims extract`)
- Apply extraction quality patterns from <EntityLink id="E902">E902</EntityLink>
- Validate on 5-10 high-importance entity pages
- Target: \>95% self-contained, \>80% sourced

### Phase 3: Migration

- Migrate high-quality claims from legacy Claims table
- Retire `/claims` UI routes (redirect to `/statements`)
- Consolidate internal dashboards

### Phase 4: Resource-First Extraction

- Extract statements from resources directly
- Route to correct entities via LLM-based relevance scoring
- Deduplication infrastructure

### Phase 5: Page Integration

- Feed verified statements into `crux content improve` pipeline
- Contradiction detection (page prose vs stored statements)
- Gap analysis ("your page covers X but has no statements about Y")
