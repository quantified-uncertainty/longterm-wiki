> **Note**: These patterns were documented during the Claims era but apply equally to Statement extraction. See <EntityLink id="E1006" name="statements-system-architecture">Statements System Architecture (E1006)</EntityLink> for the current system.

## Purpose

This document catalogs the specific ways statement extraction goes wrong. Each pattern is documented with:
- **Real examples** from the database (not hypotheticals)
- **Root cause** — why the extraction LLM produces this
- **Detection** — how to catch it programmatically
- **Proposed fix** — prompt change, post-processing, or both

The goal is to turn these patterns into a validation suite that runs on every extraction, rejecting or flagging bad claims before they enter the database. See <EntityLink id="E901" name="claims-data-quality-audit">Claims Data Quality Audit (E901)</EntityLink> for the aggregate statistics.

## Pattern 1: Context-Dependent Claims (the "orphan sentence" problem)

**Frequency:** 45.8% of claims lack entity names; 13.1% start with relative phrases.

**What it looks like:**

| Bad (context-dependent) | Good (self-contained) |
|---|---|
| "Founded in 2018" | "Kalshi was founded in 2018" |
| "The interpretability team has 40-60 researchers" | "Anthropic's interpretability team has 40-60 researchers as of 2025" |
| "Some analysts interpret the high rejection rate as excessive caution" | "Some analysts interpret Claude's high refusal rate as excessive caution by Anthropic" |
| "Research investment is high" | "Research investment in RLHF techniques has been high across major AI labs" |

**Root cause:** The LLM extracts sentences from prose that already established context in preceding paragraphs. "The interpretability team has 40-60 researchers" makes perfect sense on the Anthropic page — the reader knows which company. But the extraction prompt asks for "atomic claims" without reinforcing that each claim must stand alone.

**Detection heuristics:**
- Claim doesn't contain any known entity name or alias
- Claim starts with "The ", "This ", "These ", "Their ", "Its "
- Claim contains only pronouns ("it", "they") with no antecedent

**Proposed fix:**

Extraction prompt addition:
```
CRITICAL: Every claim must be a complete, self-contained assertion.
A reader who sees ONLY this claim (not the page, not the section heading,
not the surrounding claims) must be able to understand what it asserts and
about whom. Always include the full entity name. Never use "the company",
"the model", "the platform", or similar — use the actual name.
```

Post-processing validation: reject claims where no recognized entity name appears in the text. Return them to the LLM with: "This claim is not self-contained. Rewrite to include the entity name: [entity_id]."

## Pattern 2: Entity Misattribution

**Frequency:** 5.2% of claims (78 claims).

**What it looks like:**

A claim extracted from the Anthropic page about another entity:

| Claim | Stored Under | Should Be Under |
|---|---|---|
| "OpenAI o3 achieved state-of-the-art results on mathematics benchmarks" | anthropic | openai |
| "ChatGPT is deployed by OpenAI" | rlhf | openai |
| "Google DeepMind published the Gemini 1.5 technical report" | anthropic | deepmind |

**Root cause:** The extraction pipeline assigns ALL claims from a page to that page's entity. When the Anthropic page mentions competitors (OpenAI, DeepMind), those mentions get extracted as Anthropic claims.

**Why this is particularly damaging:** These claims actively mislead. A user querying "what do we know about Anthropic?" would see claims about OpenAI's models. The claim store becomes unreliable for the most basic use case.

**Detection heuristics:**
- Claim mentions entity Y prominently but not entity X (the assigned entity)
- Claim's `relatedEntities` includes Y but entity_id is X
- Named entity recognition on claim text identifies a different primary subject

**Proposed fix:**

Two-stage approach:
1. **Extraction prompt:** "Only extract claims that are ABOUT [entity_name]. If a sentence mentions [entity_name] in passing but is primarily about another entity, skip it."
2. **Post-extraction routing:** After extraction, check each claim: does it actually mention the assigned entity? If not, either re-route to the correct entity or discard.

**Technical challenge:** Some cross-entity claims are legitimate. "Anthropic emerged from disagreements within OpenAI" is correctly stored under both entities. The fix needs to distinguish "claim about X that mentions Y" from "claim about Y extracted from X's page."

## Pattern 3: MDX Markup Leakage

**Frequency:** 4.6% of claims (69 claims).

**What it looks like:**

```
"White House AI Czar <EntityLink id="E431" name="david-sacks">John Smith</EntityLink> announced new regulations"
"Anthropic's valuation reached <KBF entity="anthropic" property="valuation" />"
"The model achieved {/* NEEDS CITATION */} impressive results"
```

**Root cause:** The extraction LLM receives the raw MDX source (including JSX components) and copies them verbatim into claim text. The MDX is not cleaned before extraction.

**Three sub-types:**
1. **`<EntityLink>` tags** — These are arguably acceptable if we render them later, but they make claim text ugly in non-rendering contexts (search, API responses, exports)
2. **`<KBF>` canonical fact tags** — These resolve to dynamic values at render time. A claim containing `<KBF entity="anthropic" property="valuation" />` is meaningless without the KB resolution pipeline. The claim should contain the actual value.
3. **MDX comments `{/* */}`** — These should never appear in claims. They indicate the extraction pulled in editorial notes.

**Detection:**
- Regex: `/<[A-Z][a-zA-Z]*|<\/|{\s*\/\*/`
- Specifically: `/<EntityLink/, /<F\s/, /{\s*\/\*`

**Proposed fix:**

Pre-extraction cleanup: strip or resolve MDX before passing to the extraction LLM.

Before extraction, cleanMdxForExtraction() should:
  - Resolve EntityLink tags to plain text (e.g., "Anthropic")
  - Resolve F tags to actual fact values (lookup from facts YAML)
  - Strip MDX comments
  - Strip other JSX components

Note: `cleanMdxForExtraction()` already exists in `crux/claims/extract.ts` but may not handle all cases. Verify coverage.

**Decision needed:** Should `<EntityLink>` markup be preserved in claims or stripped? Current decision: preserve but add a `claim_text_plain` field for contexts that can't render MDX.

## Pattern 4: Multiple Facts Per Claim

**Frequency:** 2.4% of claims (36 claims), but likely undercounted.

**What it looks like:**

```
"Anthropic raised $7.3 billion in total 2024 funding, bringing the company's
post-money valuation to $61.5 billion, with backing from Google, Salesforce Ventures,
and Amazon."
```

This single "claim" contains at least 4 verifiable assertions:
1. Anthropic raised \$7.3B in 2024
2. This brought the valuation to \$61.5B
3. Google backed the round
4. Salesforce Ventures and Amazon also backed the round

**Root cause:** The extraction LLM mirrors the source text's sentence structure. Dense factual sentences in wiki prose get extracted as single claims.

**Why this matters:**
- Verification becomes ambiguous — is the claim "verified" if 3 of 4 facts check out but the 4th is wrong?
- Deduplication fails — another claim "Anthropic raised \$7.3B in 2024" won't match this compound claim
- Structured queries are impossible — you can't query "which companies backed Anthropic's 2024 round?" from this claim

**Detection heuristics:**
- Claim contains semicolons separating independent clauses
- Claim contains ", and" or ", with" connecting distinct facts
- Claim has more than 2 distinct numeric values
- Claim length \> 200 characters (correlated with compound claims)

**Proposed fix:**

Extraction prompt addition:
```
Each claim should contain exactly ONE verifiable assertion. If a sentence
contains multiple facts (e.g., "X raised $Y at a $Z valuation with backing
from A, B, and C"), split it into separate claims for each fact.
```

**Trade-off:** More atomic claims = more records, more verification cost, but much better queryability and dedup. The <EntityLink id="E896" name="claims-architecture-decisions">Architecture Decisions</EntityLink> document discusses this as Crux 7 (the atomicity problem).

## Pattern 5: Vague / Unverifiable Claims

**Frequency:** 1.3% explicitly vague, but the real rate is higher.

**What it looks like:**

| Claim | Problem |
|---|---|
| "Research investment is high" | What does "high" mean? Compared to what? |
| "The model showed significant improvement" | How much improvement? On what? |
| "Various organizations have expressed concern" | Which organizations? What concern? |
| "Several researchers have proposed alternatives" | Who? What alternatives? |
| "The regulatory landscape is evolving rapidly" | This is always true of everything |

**Root cause:** The source pages contain vague prose, and the extraction LLM faithfully reproduces it. The extraction prompt doesn't penalize vagueness.

**Detection heuristics:**
- Contains: "significant", "various", "several", "numerous", "many", "rapidly", "increasingly"
- Contains no specific numbers, dates, names, or measurable quantities
- Short length + hedge words = high probability of being unverifiable

**Proposed fix:**

Extraction prompt addition:
```
Skip claims that are vague or unverifiable. A good claim contains specific
names, numbers, dates, or measurable quantities. A bad claim uses words like
"significant", "various", "several", or "many" without specifics. If the
source text is vague, either make it specific (if you can from context) or
skip it.
```

**Challenge:** Some legitimately important claims are qualitative ("RLHF is the dominant alignment technique"). The fix needs to distinguish "vague" from "qualitative but specific."

## Pattern 6: Opinions Classified as Factual

**Frequency:** 1.3% (19 claims), but likely higher with less obvious cases.

**What it looks like:**

```
claimType: "factual"
claimText: "Critics argue that Anthropic's safety testing is insufficient for
           the capabilities of its most advanced models"
```

This isn't a fact — it's an attributed opinion. It should be `claimType: "evaluative"` with `claimMode: "attributed"`.

**Other examples:**
- "Some analysts perceive the pricing as aggressive" (factual → evaluative)
- "Kalshi is arguably the most innovative prediction market" (factual → evaluative)
- "The approach is considered groundbreaking by many researchers" (factual → consensus)

**Root cause:** The extraction LLM defaults to "factual" for everything. The `claimType` taxonomy has 7 values, but the model under-uses all but "factual."

**Detection heuristics:**
- Contains attribution markers: "critics", "analysts", "observers", "some argue", "widely considered", "perceived as"
- Contains evaluation language: "arguably", "innovative", "groundbreaking", "insufficient", "excessive"
- Pattern: [attribution source] + [evaluative verb] + [claim content]

**Proposed fix:**

Two approaches:
1. **Better type instructions** in extraction prompt with explicit examples for each type
2. **Post-extraction reclassification** — a second LLM pass that only classifies claim type, using the full taxonomy with examples

The second approach separates concerns (extraction vs. classification) and may be more reliable, but doubles cost.

## Pattern 7: Temporal Ambiguity

**Frequency:** Not measured directly, but pervasive.

**What it looks like:**

```
"Anthropic has approximately 1,500 employees"
```

When was this true? Is it still true? The claim has no `as_of` date.

**Worse:**
```
"Anthropic recently raised a new funding round"
```

"Recently" relative to when? The page was written in January 2026 — by March 2026, this is stale.

**Root cause:** The `as_of` field exists in the schema but is rarely populated by extraction. The extraction prompt doesn't ask for it consistently.

**Detection heuristics:**
- Contains time-relative words: "recently", "currently", "now", "this year", "last year"
- Contains quantities that are inherently volatile: employee counts, valuations, revenue
- `as_of` field is null but claim contains time-sensitive content

**Proposed fix:**

Extraction prompt should require `as_of` for any claim containing:
- Numeric quantities (financial, headcount, metrics)
- Time-relative language
- Words implying recency or currency

Post-extraction: flag claims with volatile-looking content but no `as_of`.

## Pattern 8: Claims About Claims (meta-assertions)

**Frequency:** Low but insidious.

**What it looks like:**

```
"Research has shown that alignment techniques can be effective"
"Studies suggest that scaling may lead to emergent capabilities"
"Experts generally agree that AI safety is important"
```

These aren't claims about the world — they're claims about what other people claim about the world. They're meta-assertions that can't be directly verified against a single source.

**Root cause:** Wiki pages often summarize research consensus using "research shows" / "studies suggest" framing. The extraction LLM faithfully extracts these summary statements.

**Why this matters:** These claims have a hidden verification challenge. "Research has shown X" can only be verified by checking multiple studies — it's really a consensus claim dressed up as a factual one.

**Proposed fix:** Classify these as `claimType: "consensus"` and require multiple source references. The extraction prompt should distinguish "Source X says Y" (attributed, single source) from "Research shows Y" (consensus, needs breadth).

## Pattern 9: Truncation Artifacts

**Frequency:** \~2 instances in Sprint 1 sample, likely more at scale.

**What it looks like:**

```
"Anthropic's approach to constitutional AI involves training models using a set of"
```

The claim is cut off mid-sentence.

**Root cause:** LLM output truncation (context window limits or max_tokens hit during extraction), or the section text was truncated before being sent to the LLM.

**Detection:** Claims ending without terminal punctuation (no period, question mark, or closing quote).

**Proposed fix:** Post-extraction validation: reject claims that don't end with terminal punctuation. These are always extraction errors, never valid claims.

## Pattern 10: Tautological Claims

**Frequency:** Estimated 3-5%, based on Sprint 1 "trivially obvious" findings.

**What it looks like:**

```
"Kalshi is a regulated prediction market exchange" (on the Kalshi page)
"RLHF stands for Reinforcement Learning from Human Feedback" (on the RLHF page)
"Anthropic is an AI safety company" (on the Anthropic page)
```

These are true but add zero value. They restate the entity's defining property.

**Root cause:** Wiki pages typically start with a definitional sentence. The extraction LLM faithfully extracts it.

**Detection heuristics:**
- Claim is a definitional statement ("X is a Y")
- Claim text closely matches the entity's `description` field in the database
- Claim contains only information already in the entity's metadata

**Proposed fix:**

Extraction prompt:
```
Skip claims that merely define what the entity is. "Kalshi is a prediction
market" is not a useful claim — everyone reading the Kalshi page already knows
this. Focus on specific, non-obvious assertions.
```

## Pattern 11: Source Quote Mismatch

**Frequency:** Not systematically measured. Known to exist from Sprint 1 "Minor issues 85%" finding on Kalshi.

**What it looks like:**

```
claimText: "Kalshi's fee structure incentivizes liquidity provision while
            remaining competitive with traditional sportsbook vigorish"
sourceQuote: "Kalshi charges transaction fees on trades"
confidence: "verified" (partial match)
```

The source says "charges fees." The wiki page (and claim) adds an analytical interpretation about "incentivizing liquidity" that doesn't appear in the source.

**Root cause:** The wiki page embellishes beyond what sources actually say. The extraction faithfully captures the embellished version. Verification then finds partial support because the underlying fact is there, but the analytical framing isn't.

**Why this matters:** This is the subtlest and most dangerous failure mode. The claim looks verified (it has a source quote), but the actual assertion goes beyond what the source supports.

**Proposed fix:** This is fundamentally a page quality issue, not an extraction issue. The extraction is working correctly — it's capturing what the page says. The fix is:
1. Flag claims where `confidence: "verified"` but the source quote doesn't contain the claim's key assertion
2. The page improvement pipeline should flag analytical claims that lack source support
3. Consider `claimMode: "editorial"` for claims that represent the wiki's own analysis rather than sourced facts

## Pattern 12: Quantity Without Context

**Frequency:** Common in financial/metric claims.

**What it looks like:**

```
"Revenue grew 70% year-over-year"
"The model achieved 86.4% on MMLU"
"User count reached 100 million"
```

These have numbers but lack crucial context:
- 70% growth from what base? \$1M → \$1.7M is different from \$1B → \$1.7B
- 86.4% on MMLU — is that good? What did competitors score?
- 100M users — as of when? Monthly active? Total signups?

**Root cause:** The source pages provide context in surrounding sentences, but the extracted claim loses it.

**Proposed fix:** For quantitative claims, the extraction prompt should require:
- The `as_of` date
- The `value_numeric` field (machine-readable)
- A `measure` field clarifying what's being measured
- Sufficient context in the claim text for standalone interpretation

## Pattern 13: Extraction Without Reasoning Trail

**Frequency:** ~100% of current claims (the system doesn't generate traces yet).

**What it looks like:**

A claim exists in the database with a verdict ("verified") and a source quote, but no record of *how* the extraction LLM connected the source to the claim — what inference was involved, what alternatives were considered, or why this particular characterization was chosen over others.

| Claim | Source Quote | What's Missing |
|---|---|---|
| "Anthropic's Constitutional AI represents a fundamentally different approach from RLHF" | "we use AI feedback to evaluate model outputs rather than human feedback" | Why is "fundamentally different" justified vs. "a variant of" or "an evolution of"? |
| "Kalshi is the leading regulated prediction market" | "the largest US-regulated exchange" | How did "largest" become "leading"? Are they equivalent? |
| "Research investment in RLHF has been high across major AI labs" | (no source quote) | What evidence supports "high"? Compared to what baseline? |

**Root cause:** The extraction prompt asks for claims and optionally source quotes, but never asks the LLM to explain its reasoning. The LLM makes implicit judgment calls (interpreting "largest" as "leading," adding the word "fundamentally") without documenting them. These undocumented inference steps are where science-washing occurs — the claim drifts from what the source actually says, and there's no audit trail to catch it.

**Why this is especially damaging for AI verification:** An AI re-verifier checking "is this claim supported by this source?" must reconstruct the reasoning from scratch. Without a trace, it can't tell whether the extraction LLM intended a direct assertion (source says X, claim says X) or an interpreted assertion (source says X, claim infers Y from X). The verifier's job is fundamentally harder.

**Detection heuristics:**
- Claims with `inferenceType: null` (no trace at all)
- Claims where claim text uses qualitative language ("leading," "significant," "fundamentally") not present in source quote
- Claims classified as evaluative/causal/consensus but with no `inferenceStep` explaining the evaluation/causation/consensus

**Proposed fix:**

This is not a post-processing fix — it requires changes to the extraction pipeline:

1. **During extraction:** Add to the prompt: "For each claim, indicate the inference type: 'direct' if the source explicitly states this, 'derived' if you inferred it from the source, or 'editorial' if this is the wiki's own analysis. For derived and editorial claims, briefly explain the reasoning."

2. **Post-extraction validation:** Flag claims where `inferenceType` is `derived` or `editorial` but `inferenceStep` is empty. These are the highest-risk claims for science-washing.

3. **Long-term:** The <EntityLink id="E903" name="reasoning-traces-architecture">Reasoning Traces Architecture (E903)</EntityLink> proposes a full trace model with tiered depth — minimal traces for direct assertions, structured traces for derived claims, full traces for editorial claims.

## Compound Issue: The Quality Cascade

These patterns don't occur independently. A single bad claim often exhibits 3-5 issues simultaneously. The worst claims in the database (7-8 issues each) show a cascading pattern:

```
1. Source page has vague prose about another entity
2. Extraction captures it faithfully (but it's vague AND about the wrong entity)
3. No entity name (because the page established context paragraphs ago)
4. No source (because the page section had no footnotes)
5. No relatedEntities (because the extraction didn't populate the field)
6. Starts with "The" or "This" (relative phrase)
7. Tagged as "factual" (when it's really evaluative)
```

**Implication:** Fixing individual patterns in isolation has diminishing returns. The highest-impact fix is improving extraction to produce self-contained claims in the first place (Pattern 1), which automatically reduces the incidence of Patterns 2, 4, 5, 7, and 12.

## Proposed Validation Pipeline

Based on these patterns, every extracted claim should pass through these checks before database insertion:

```
1. SELF-CONTAINED: Contains a recognized entity name? (Pattern 1)
2. CORRECTLY ATTRIBUTED: Mentions its assigned entity? (Pattern 2)
3. CLEAN TEXT: No MDX markup (except EntityLink)? (Pattern 3)
4. ATOMIC: Contains exactly one assertion? (Pattern 4)
5. SPECIFIC: No vague quantifiers without specifics? (Pattern 5)
6. CORRECTLY TYPED: claimType matches content? (Pattern 6)
7. TEMPORALLY GROUNDED: Volatile claims have as_of? (Pattern 7)
8. COMPLETE: Ends with terminal punctuation? (Pattern 9)
9. NON-TAUTOLOGICAL: Not just restating entity definition? (Pattern 10)
10. CONTEXTUALLY COMPLETE: Quantities have sufficient context? (Pattern 12)
```

Claims failing checks 1, 2, 3, 8, or 9 should be **rejected** (these are always errors).
Claims failing checks 4, 5, 6, 7, 10, or 12 should be **flagged** (may need human judgment).

**Cost estimate:** Running this validation as a second LLM pass costs ≈\$0.001-0.005 per claim. For 1,514 claims, that's \$1.50-7.50 — trivial compared to the cost of bad data persisting.

## Investigation Needed

Several open questions that require empirical testing:

1. **Does `cleanMdxForExtraction()` handle all markup types?** The function exists in `crux/claims/extract.ts` but Pattern 3 shows markup still leaking through. Need to audit its coverage.

2. **What's the actual rate of Pattern 11 (source quote mismatch)?** Running verification on all claims would reveal how many "verified" claims actually have weak source support.

3. **Does two-pass extraction (extract then validate) outperform single-pass?** Sprint 2 of the <EntityLink id="E897" name="claims-system-development-roadmap">Development Roadmap</EntityLink> is designed to test this.

4. **Can pg_trgm similarity (PR #1149) catch near-duplicate claims effectively?** The trigram similarity threshold needs empirical tuning — too aggressive merges distinct claims, too loose misses real duplicates.

5. **What's the right prompt for concept pages?** The 69% usefulness rate on concept pages is the biggest quality gap. These pages need fundamentally different extraction instructions — possibly extracting "key propositions" rather than "factual claims."
