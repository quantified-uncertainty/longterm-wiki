## Context

This document synthesizes a design session exploring how to significantly expand the claims system, with an initial focus on **people and organizations** — where claims are concrete, factual, and directly verifiable — rather than high-level conceptual topics. It is a companion to <EntityLink id="claim-first-architecture">Claim-First Architecture (E893)</EntityLink>.

The immediate trigger: we want to spend significant effort dramatically improving wiki pages for people and organizations using Claims + Resources as the connective tissue. The Kalshi page is the current exemplar of the end state. The question is how to get there systematically across ~100+ entity pages, and what infrastructure changes are needed to support that.

---

## Current System Snapshot

Three overlapping layers exist today:

| Layer | Storage | Scope | Limitations |
|---|---|---|---|
| **Facts** | YAML (`data/facts/*.yaml`) | 18 entities, numeric-only | Only quantitative; no verification; manual authoring |
| **Claims** | PostgreSQL (`claims` table) | Any wiki page | Extracted from MDX prose; sparse schema; LLM-quality variable |
| **Resources** | YAML (`data/resources/*.yaml`) | ≈5,000+ entries | Static; sparse metadata; passive citation targets |

The core problem: **resources are treated as passive citations**, not as active knowledge sources. Claims are extracted *from* wiki pages that cite resources, but never extracted *from* the resources themselves. The information flow is one-directional, when it should be bidirectional.

---

## Key Architectural Decisions Made

### 1. Resources → PostgreSQL

YAML is no longer the right storage for resources at scale (`web-other.yaml` is 39K lines, 4200+ entries). The migration path:

- YAML remains a valid **import format** for seeding
- PostgreSQL is the **live database** going forward
- The Next.js build reads from a Postgres-generated snapshot instead of raw YAML
- CLI creates resources via `crux resource add <url>`, not YAML editing
- Target: before the claim store exceeds 5k records, resources are fully in Postgres

### 2. Resources Are a Distinct Entity Type (Not Wiki Pages)

Resources and wiki pages overlap in interesting ways (both can have claims extracted from them, both represent knowledge objects) but are fundamentally different:

- **Provenance**: Wiki pages are *our* authored analysis. Resources are *someone else's* content. Claims extracted from a resource are the original author's claims, not ours. This distinction matters for epistemic attribution.
- **Ephemerality**: Resources can 404, get paywalled, or be corrected. Wiki pages are ours to maintain.
- **Type-specific metadata**: What a paper needs (DOI, citation count, peer review) is incompatible with what a video needs (channel, duration, transcript) vs. a news article (outlet, time-sensitivity). A single schema can't serve all types well.
- **Importance**: Most resources (random news citations) will never merit a wiki page. A small fraction (~5%) are foundational texts that deserve dedicated analysis pages.

Resources get a `wiki_page_id` field: if a resource is important enough to have a dedicated wiki page about it (e.g., "Attention Is All You Need", "AGI Ruin: A List of Lethalities"), that link is explicit.

### 3. Claims Have Multiple Sources (Join Table)

A single claim can be attested by multiple independent resources. "Anthropic's valuation is \$380B" appears in Reuters, TechCrunch, Bloomberg, and Anthropic's own press release simultaneously. Multiple sources are the normal case for important claims, not an edge case.

```text
-- claim_sources join table (new; claims currently use footnote_refs string)
claim_id      bigint  → claims.id
resource_id   text    → resources
source_quote  text    -- excerpt from THIS source supporting the claim
is_primary    bool    -- which source to show first in UI
added_at      timestamp
```

Multi-source corroboration is how claim credibility is computed — 5 independent high-credibility sources agreeing is epistemically different from 1 preprint.

### 4. Claims and Facts Are Related but Distinct

Facts are the **promoted, canonical, display-ready** layer for quantitative data. Claims are the **broader epistemic** layer covering all assertion types. The relationship:

```
Claim (any assertion) ──── fact_id ────→ Fact (canonical numeric, rendered via <F>)
```

A fact like "Anthropic valuation is \$380B" IS a claim — the promoted version. The underlying claim has multi-source provenance, verification history, and a `fact_id` linking to the YAML entry. Facts are a curated subset of the claim space.

The claims schema should borrow from facts:
- **`as_of`** (time-indexing — currently missing from claims entirely)
- **`measure`** (taxonomy linking quantitative claims to fact-measures, enabling timeseries)
- **`value_numeric`** / **`value_low`** / **`value_high`** (machine-readable values for quantitative claims)
- **footnote refs** (currently stored as comma-separated string in `unit` column) → replaced by the `claim_sources` join table above

### 5. Every Claim Must Trace to a Source

Non-negotiable constraint: every claim in the system must have at least one entry in `claim_sources`. Unsourced claims should not be insertable (enforced at the API layer). The current `confidence: unsourced` status is for claims extracted from wiki pages where the prose makes an assertion with no footnote — those should be flagged for cleanup, not normalized as an acceptable state.

### 6. Faceted Classification Over Tags

Five orthogonal facets, fixed vocabulary enforced at write time:

| Facet | Values | Purpose |
|---|---|---|
| `claim_type` | factual, historical, causal, evaluative, predictive, normative, consensus | Epistemic type |
| `claim_mode` | endorsed, attributed | Wiki asserts vs. person X claims |
| `domain` | financial, personnel, research, governance, product, safety, legal, strategic | Topic area |
| `confidence` | verified, unverified, unsourced, contradicted | Trust state (current code has verified/unverified/unsourced; `contradicted` is proposed addition) |
| `volatility` | stable, annual, quarterly, unpredictable | How fast it changes |

Tags are explicitly avoided — they degrade past ~1,000 items without strict governance. Optional lightweight `cluster` slugs handle natural groupings (e.g., `anthropic-series-g`) without requiring a taxonomy.

## Worked Examples

### Example A: Kalshi (Commercial Entity Page)

Kalshi is the current exemplar with 192 claims, 37 sources, inline verification. Here's what the claim store looks like for a few representative claims:

```
claim: "Kalshi's December 2025 post-money valuation was $11 billion"
  claim_type:   factual
  claim_mode:   endorsed
  domain:       financial
  as_of:        2025-12
  value_numeric: 11000000000
  measure:      valuation
  volatility:   annual
  confidence:   verified
  cluster:      kalshi-financials-2025
  sources:
    - resource_id: [techcrunch-article-id]
      source_quote: "the company reached an $11 billion valuation..."
      is_primary: true

claim: "Kalshi charges taker fees ranging from 0.07% to 7%"
  claim_type:   factual
  claim_mode:   endorsed
  domain:       financial
  volatility:   stable
  confidence:   verified
  sources:
    - resource_id: [kalshi-fee-schedule-id]
      source_quote: "Taker fees: 0.07% minimum, 7% maximum"
      is_primary: true

claim: "Kalshi faces a class action lawsuit alleging market-maker advantages"
  claim_type:   factual
  claim_mode:   endorsed
  domain:       legal
  as_of:        2024
  volatility:   unpredictable
  confidence:   unverified
  sources:
    - resource_id: [court-filing-id]
      is_primary: true
```

### Example B: Anthropic (Major Organization Page)

For an org page, claims span financial, personnel, research, and strategic domains — and many are already captured as Facts. The integration point:

```
# This claim IS the fact data/facts/anthropic.yaml → "6796e194"
claim: "Anthropic's post-money valuation was $380 billion as of February 2026"
  claim_type:   factual
  claim_mode:   endorsed
  domain:       financial
  as_of:        2026-02
  value_numeric: 380000000000
  measure:      valuation
  volatility:   annual
  confidence:   verified
  fact_id:      "6796e194"     ← links to canonical fact entry
  cluster:      anthropic-series-g
  sources:
    - resource_id: 8e3ff50b9ef2a1a8   ← the Series G Reuters article
      source_quote: "...closing a $30 billion Series G at a $380 billion post-money valuation"
      is_primary: true
    - resource_id: [techcrunch-id]
      source_quote: "Anthropic's new valuation of $380B..."
      is_primary: false

# A personnel claim — no corresponding Fact
claim: "Anthropic's interpretability team has 40-60 researchers as of 2025"
  claim_type:   factual
  claim_mode:   endorsed
  domain:       personnel
  as_of:        2025
  value_low:    40
  value_high:   60
  measure:      interpretability-team-size
  volatility:   annual
  confidence:   verified
  fact_id:      "28a43350"    ← also has a canonical fact entry
  sources:
    - resource_id: 4507d36fc38ca05d
      is_primary: true

# A strategic claim — evaluative, no numeric value
claim: "Anthropic's Constitutional AI approach differs from RLHF by using AI feedback rather than human feedback"
  claim_type:   factual
  claim_mode:   endorsed
  domain:       research
  volatility:   stable
  confidence:   verified
  sources:
    - resource_id: [constitutional-ai-paper-id]
      source_quote: "...rather than training a separate reward model from human feedback..."
      is_primary: true

# An attributed claim — not what WE assert, what Dario asserts
claim: "Dario Amodei believes Claude will be performing at the level of a Nobel Prize winner within 1-2 years"
  claim_type:   predictive
  claim_mode:   attributed
  attributed_to: dario-amodei      ← entity_id of the person making the claim
  domain:       research
  as_of:        2025-11
  volatility:   unpredictable
  confidence:   verified           ← verified that he said this, not that it's true
  sources:
    - resource_id: [interview-resource-id]
      source_quote: "I think within one to two years..."
      is_primary: true
```

### Example C: Neel Nanda (Person Page)

Person pages have a different claim profile — biographical facts, affiliations, stated positions, research contributions. This is where the system currently has the biggest gap: the `data/experts.yaml` and person MDX pages have no structured claim layer at all.

```
# Biographical — stable facts
claim: "Neel Nanda studied Mathematics at Trinity College, Cambridge"
  claim_type:   historical
  claim_mode:   endorsed
  domain:       personnel
  volatility:   stable
  confidence:   verified
  sources:
    - resource_id: [neel-nanda-bio-source]
      source_quote: "Mathematics at Trinity College, Cambridge"
      is_primary: true

claim: "Neel Nanda worked at Anthropic before joining Google DeepMind"
  claim_type:   historical
  claim_mode:   endorsed
  domain:       personnel
  as_of:        2022     ← when this transition happened
  volatility:   stable
  confidence:   verified
  related_entities: [anthropic, deepmind]   ← both entities are relevant
  sources:
    - resource_id: [linkedin-or-bio-source]
      is_primary: true

# Research contribution — has a specific resource (the paper)
claim: "Neel Nanda created TransformerLens, an open-source library for mechanistic interpretability research"
  claim_type:   historical
  claim_mode:   endorsed
  domain:       research
  as_of:        2022
  volatility:   stable
  confidence:   verified
  sources:
    - resource_id: [transformerlens-github-id]
      source_quote: "TransformerLens is a library for doing mechanistic interpretability..."
      is_primary: true

# A stated position — attributed, not endorsed
claim: "Neel Nanda views mechanistic interpretability as necessary for evaluating AI safety properties"
  claim_type:   evaluative
  claim_mode:   attributed
  attributed_to: neel-nanda
  domain:       safety
  volatility:   stable
  confidence:   verified
  sources:
    - resource_id: [alignmentforum-post-id]
      source_quote: "...understanding model internals is necessary before we can..."
      is_primary: true

# A quantitative contribution claim
claim: "Neel Nanda's '200 Concrete Open Problems in Mechanistic Interpretability' identified research directions as of 2022"
  claim_type:   historical
  claim_mode:   endorsed
  domain:       research
  as_of:        2022
  volatility:   stable
  confidence:   verified
  sources:
    - resource_id: [200-open-problems-resource-id]
      is_primary: true
```

The person page use case reveals a gap: we need a claim_subject field that clarifies "this claim is *about* Neel Nanda" (biographical) vs. "this claim is *by* Neel Nanda" (attributed position). The `attributed_to` field handles the latter; the former defaults to `entity_id`. But for claims like "Neel Nanda's paper argues X" — the subject is Neel Nanda, the attribution is also Neel Nanda, and the resource is the paper. These three pointers need to stay distinct.

## Key Cruxes

These are the unresolved questions where the right answer significantly changes the architecture:

### Crux 1: Claims as Global vs. Per-Page

**Option A (Per-page):** A claim belongs to exactly one wiki page. The same underlying fact appears as separate claim records on the Anthropic page and the Dario Amodei page. Deduplication is not attempted.
- *Pro:* Simple. Each page's claim set is self-contained. No dedup infrastructure needed.
- *Con:* Volume inflates 3-5x. Cross-entity queries are polluted. "How many distinct claims do we have about Anthropic?" becomes unanswerable.

**Option B (Global):** Claims are deduplicated. "Anthropic's valuation is \$380B" is one record regardless of how many pages reference it.
- *Pro:* Clean cross-entity queries. Better metrics. Avoids explosion.
- *Con:* Dedup is hard (requires semantic similarity, not string matching). The boundary between "same claim" and "similar but distinct claim" is fuzzy. Retroactive dedup is very painful.

**Current system is de facto Option A.** Before scaling, this choice needs to be explicit.

### Crux 2: Extraction Direction

**Option A (From wiki pages → claims):** The current approach. We author a wiki page, then extract claims from it.
- Matches how the wiki is authored today
- Claims reflect our editorial choices, not raw source content
- Claims can be stale when pages aren't updated

**Option B (From resources → claims):** Extract claims from source materials directly, then use those to inform/generate wiki pages.
- Resources are the ground truth; wiki pages are derived
- Requires the claim-first architecture (E893)
- Much higher value, much harder to build

**Option C (Both, with linkage):** Extract from pages AND resources; link them where the same claim appears in both.
- Most powerful but requires dedup infrastructure from Crux 1

The focus for people/org pages suggests Option B is the right long-term direction — the most valuable claims about Neel Nanda come from papers, interviews, and bio pages, not from our wiki prose about him.

### Crux 3: The Fact Integration Question

Should facts eventually be a type of claim, or should they remain a separate system with a bridge (`fact_id`)?

**Merge:** One unified store. Facts are claims with `is_canonical: true` and `value_numeric` populated. The `<F>` component reads from the claims table.
- *Pro:* Single system to maintain. Provenance unified. Verification applies to facts too.
- *Con:* Facts have compute expressions, formatting rules, build-time derivation. Migrating this to Postgres is a significant project. The `<F>` component would need to query Postgres at build time.

**Bridge:** Facts stay in YAML, claims in Postgres. `fact_id` links them. Facts are the curated promotion of important claims.
- *Pro:* No migration needed. Existing `<F>` infrastructure unchanged. Iterate on claims independently.
- *Con:* Two systems to maintain. Provenance split. A fact can drift from its corresponding claim without detection.

The bridge is the pragmatic near-term choice. Merge is the right long-term direction if the claim system proves its value.

### Crux 4: Attribution vs. Endorsement at Scale

The `claim_mode: attributed | endorsed` distinction is clear in theory:
- *Endorsed:* "Anthropic's valuation is \$380B" — the wiki asserts this
- *Attributed:* "Dario Amodei believes Claude will be Nobel-level within 1-2 years" — we report that he said this

In practice, the boundary is fuzzy. "Anthropic has strong safety commitments" — is that endorsed (our assessment) or attributed (Anthropic says so)? How do we handle cases where we partially agree with an attributed claim?

The question is whether the `claim_mode` field adds enough clarity to be worth the classification overhead, or whether it creates a pseudo-distinction that people apply inconsistently.

### Crux 5: Verification Economics

Full verification (checking claim against source text via LLM) costs money and time. At scale:
- 700 pages × ~100 claims/page = 70,000 claims
- ≈\$0.01-0.05 per verification = \$700-3,500 for a full pass
- Sources change, so verification needs to be re-run periodically

Options:
1. **Verify everything lazily** (verify when the source is re-crawled)
2. **Verify selectively** (only verify claims on high-importance pages, or claims with `volatility: stable`)
3. **Verify on insertion only** (verify once, mark permanently)
4. **Don't verify most claims** (accept that most claims are `confidence: unverified` indefinitely)

Option 4 is where the system currently ends up by default. The `confidence` field loses meaning if >90% of claims are `unverified`.

### Crux 6: Public vs. Internal Exposure

Currently claims are shown on `/wiki/[id]/data` (not the main page), with verification badges surfaced inline on main pages for citation quotes. As the claim store grows:
- Do claims become a first-class public-facing feature?
- Is the `/data` sub-page the right UX, or should it be integrated into the main page?
- For person pages, would showing "87 verified claims" alongside the prose be useful or distracting?

The Kalshi page shows verification inline on the main page (Accurate 100% badges). That's powerful for commercial entities where readers care about factual accuracy. For a researcher biography, it might feel clinical.

### Crux 7: The Atomicity Problem

"Neel Nanda studied Mathematics at Trinity College, Cambridge" — is this one claim or three?
1. "Neel Nanda studied Mathematics"
2. "Neel Nanda attended Trinity College"
3. "Neel Nanda was at Cambridge University"

High atomicity: each claim is trivially verifiable but the system has 3x more records that individually mean less. Low atomicity: claims are easier to read but harder to query precisely (you can't ask "who studied at Cambridge?" if the location is embedded in a compound claim).

The extraction LLM will choose inconsistently. Without enforcement tooling, atomicity degrades quickly to whatever the model produces.

### Crux 8: When Does a Resource Get a Wiki Page?

Some resources are foundational enough that we want a dedicated wiki page about them (not just using them as citations). The Alignment Forum post "AGI Ruin: A List of Lethalities" arguably merits a wiki page. A Reuters article about a funding round does not.

Proposed criteria (all should be met):
- Cited by ≥3 wiki pages AND
- Is a primary source (not secondary reporting) AND
- Makes a thesis-level argument OR presents original data AND
- Has been widely referenced in the AI safety discourse

This would cover papers, major blog posts, and books — but not news articles. The practical question is whether we enforce this as a gate or let editors add wiki pages for resources ad-hoc.

### Crux 9: Claim Clustering Without Hierarchy

Flat `cluster` slugs solve the "show me all claims from the Anthropic Series G event" use case. But clusters will accumulate. 500 clusters with no structure is just a different kind of tag soup.

Adding a `cluster_type` enum (`event`, `funding-round`, `regulatory-case`, `research-debate`, `product-launch`) would give enough structure to browse clusters without requiring a full hierarchy. The question is whether to add this now or later.

### Crux 10: The Cold Start Problem

The claim system delivers value proportional to coverage. A claim store with 200 claims across 5 pages is interesting. 70,000 verified claims across 700 pages is genuinely powerful. But getting from 200 to 70,000 requires sustained investment with limited visible return in the middle.

The forcing function needs to exist *before* coverage is sufficient. Options:
- Require claims for high-importance pages (enforced by quality scoring)
- The `<Claim id="...">` component creates a direct incentive (can't use it if claims don't exist)
- Internal dashboards that make low-coverage pages visually obvious (shame-driven)
- Automated extraction as part of the auto-update pipeline (zero marginal cost per page)

## Red Team: How This Goes Wrong

### 1. LLM Extraction Quality Collapses on Non-Factual Content

Kalshi is ideal for claims extraction — dense, factual, measurable. Person pages and conceptual org pages are different. "Neel Nanda's research approach emphasizes circuit discovery" is not a clean verifiable claim — it's a characterization. LLMs will produce vague, redundant, or tautological claims at high rates for prose that isn't primarily factual.

**Mitigation:** Before investing in infrastructure, manually score extraction output on 5 person pages and 5 conceptual pages. If claim quality is poor, fix the prompt first. Don't build around bad extraction.

### 2. Deduplication Debt Accumulates Silently

If Option A (per-page) is chosen, the same claim appears on dozens of pages. "Anthropic was founded in 2021" exists as a separate record on the Anthropic page, the Dario Amodei page, the Claude page, the interpretability page... At 50k records, half are semantic duplicates, but there's no clean way to retroactively deduplicate without breaking page-level views.

**Mitigation:** Make the dedup strategy explicit before insertion volume exceeds 5k records. Either commit to global dedup now (requires embedding infrastructure) or explicitly accept per-page duplication and design around it.

### 3. The "Two Systems Forever" Trap

YAML resources + Postgres claims + YAML facts = three data stores that partially overlap and have no single authority. The plan to migrate "eventually" never completes because there's always more urgent work. Editors update YAML; the Postgres data is stale; nobody trusts either.

**Mitigation:** Set a concrete migration milestone with a deadline. Resources in Postgres before the claim count exceeds 5k. Facts in Postgres before the claim count exceeds 20k. Enforce via build failure if YAML is modified after the cutover date.

### 4. Verification Becomes Meaningless

With automated extraction producing hundreds of claims per page, and verification costing real money, `confidence: unverified` becomes the permanent state for most claims. The distinction between `unverified` (haven't checked yet) and `unsourced` (no source exists) disappears. The dashboard shows "45,000 unverified claims" indefinitely.

**Mitigation:** Rename `unverified` to `pending` at minimum. Better: only extract claims you intend to verify within 30 days. Prioritize by page importance and claim volatility.

### 5. The Synthesis Pipeline Gets Deferred Forever

The highest-value outcome is: claim store → auto-generate/improve wiki page sections. But this is the hardest part, so it gets perpetually deferred. The claim store becomes a read-only audit trail that nobody uses when writing new content, because it's easier to just ask Claude to write from scratch.

**Mitigation:** Build `crux claims synthesize <page-id>` early — even a rough version. The forcing function is showing editors that the stored claims can produce better prose than starting from scratch.

### 6. The Ontology Creeps

Start with 6 claim types, 8 domain values. In 6 months, someone adds `claim_type: definitional` for glossary entries, `domain: infrastructure` for technical claims, `domain: communications` for PR claims. The taxonomy doubles. Browsing by facet becomes useless.

**Mitigation:** Lock the taxonomy as an enum with a PR-gated change process. The first version doesn't need to be perfect — it needs to be stable. Extend only with demonstrated need across multiple pages, not for one-off cases.

### 7. Multi-Source Becomes Citation Spam

The `claim_sources` join table is the right design, but without governance, every news article that mentions a fact gets attached as a source. "Anthropic's valuation" gets 40 source entries — every tech blog that covered the funding round. The table balloons; the "corroboration" signal is lost in noise.

**Mitigation:** Distinguish `corroborating` (adds no new information) from `primary` (original reporting) sources. Only primary sources get full treatment. Corroborating sources increment a counter. Or: cap sources per claim at 5.

### 8. Person Page Claims Violate Privacy Norms

For researchers and public figures, claims about their biographical facts, stated views, and affiliations are mostly fine. But the system could easily accumulate claims about salary estimates, personal relationships, health, or other sensitive information that shouldn't be in a structured queryable database even if the information is technically public.

**Mitigation:** Add a `claim_sensitivity` field or a domain blocklist for person entities. No claims in domains like `personal`, `medical`, `financial-personal`. Only `personnel`, `research`, `stated-positions`, `affiliations` for person entities.

### 9. Schema Lock-In Before the Model Is Understood

The schema proposed above assumes claims link to pages and resources. What if the right primary entity for a claim is a *fact* (linking two facts together), or a *relationship* between two entities, or a *position on a crux*? Once 50k claims are in the database, changing the core entity model requires a painful migration.

**Mitigation:** Run a "can I answer this?" test with 15 specific queries before finalizing the schema. Include questions like: "What claims connect Anthropic and Google?" "What claims has Neel Nanda made about interpretability vs. alignment?" "What claims were true in 2023 but are now contradicted?" If the schema can't answer cleanly, change it now.

### 10. Value Never Materializes for Editors

The internal claims dashboard exists. The `/wiki/[id]/data` page shows claims. But page *editors* — the humans and agents who improve wiki pages — never consult the claim store when writing. They open the MDX file, run `crux content improve`, and the improvement pipeline ignores the claim store entirely. After 2 years, the claim store is comprehensive but has no impact on page quality.

**Mitigation:** Integrate claim lookups into `crux content improve`. Before improving a page, fetch its stored claims and include them in the improvement prompt as "verified facts about this entity." The improvement should explicitly reference stored claims rather than re-deriving them from scratch.

## Phased Delivery Plan

The goal: provide visible value quickly while building toward the full vision. Each phase has a clear deliverable and a "stop here if it's not working" checkpoint.

### Phase 0: Validate Extraction Quality (Before Any Schema Changes)
- Run extraction on 10 diverse pages: 3 org pages, 3 person pages, 2 conceptual pages, 2 event pages
- Manually score every extracted claim: useful / vague / wrong / duplicate
- Target: >60% useful claims before proceeding
- If &lt;60%: fix extraction prompt, re-evaluate. Do not invest in infrastructure yet.

### Phase 1: Get 5 Pages to Kalshi Quality
- Pick 5 high-importance org/person pages (e.g., Anthropic, OpenAI, Neel Nanda, Paul Christiano, MIRI)
- Run extract + verify + manual review for each
- The `/wiki/[id]/data` page should show comprehensive, well-sourced claims
- Deliverable: a demo showing 5 pages with 50-150 verified claims each

### Phase 2: Schema Hardening (Small, Targeted Changes)
- Add `as_of` to claims (non-nullable for new insertions)
- Add `claim_mode: attributed | endorsed`
- Add `attributed_to` field
- Rename `value` column (currently stores section name) to `section` and `unit` column (currently stores comma-separated footnote refs) to `footnote_refs` — these are the correct semantic names; `unit` and `value` were repurposed at inception
- Add `claim_sources` join table; migrate existing single-source records
- Add `value_numeric`, `value_low`, `value_high` for quantitative claims
- Deploy each change separately with data migrations. No big-bang schema overhaul.

### Phase 3: Resources in PostgreSQL
- Build `crux resource add <url>` command
- Migrate resources YAML → Postgres (automated import script)
- Update build pipeline to read resources from Postgres snapshot
- Freeze YAML files (validation error if modified)

### Phase 4: Extraction FROM Resources
- `crux claims extract-resource <resource-id>` command
- Run on the top 50 most-cited resources
- Show extracted claims on `/source/[id]` as "Claims Made by This Source"
- Distinguish visually from "Claims That Cite This Source"

### Phase 5: The Synthesis Pipeline
- `crux claims synthesize <page-id>`: fetches stored claims, generates suggested prose additions
- Initially: just lists claims that aren't yet represented in the page prose
- Later: drafts actual MDX sections referencing claims by ID
- Integrate into `crux content improve` as a pre-pass

### Stop Criteria

At each phase end, ask: are editors actually using the output? Is the claim store being consulted when pages are written? Is the data quality high enough that stored claims can be trusted without manual review?

If the answer is no at Phase 2, stop. Fix quality before adding volume. A small, trusted claim store is more valuable than a large, unreliable one.
